# SQL rewrite anti-patterns (DWS)

用于在“准备改写 SQL”时快速筛查高收益低风险问题，避免一上来就做高成本结构改造。

## 1) 高优先级反模式（先改）

### A. `SELECT *`
- 症状：扫描与回传列远超实际使用列，CN/DN 网络与 CPU 开销增大。
- 建议：仅保留业务所需列，尤其在宽表场景优先处理。

### B. 过滤列函数包裹
- 症状：谓词不 SARGable，分区裁剪/索引命中受损。
- 反例：`WHERE date_trunc('day', ts_col) = '2026-01-01'`
- 改写思路：将函数转移到常量侧或改范围过滤。

### C. 隐式类型转换
- 症状：执行计划出现不必要类型转换，行数估计偏差。
- 建议：JOIN 键、过滤值类型显式对齐。

## 2) 中优先级反模式（按证据触发）

### D. 大 `IN (...)` 列表
- 症状：优化器处理复杂、执行计划膨胀。
- 建议：改为 `EXISTS` / JOIN / 小表承载条件集。

### E. 过滤后置
- 症状：先 JOIN/聚合，再过滤，导致输入行数过大。
- 建议：尽量前移过滤，缩小后续算子输入。

### F. 不必要 `DISTINCT` / 重复排序
- 症状：`Sort` / `Aggregate` 成为热点。
- 建议：仅保留业务必需的去重与排序。

## 3) 改写输出规范

每条 rewrite 建议必须附：

1. 原写法与改写写法；
2. 语义风险（是否需业务确认）；
3. `EXPLAIN*` 对比验证动作；
4. 回滚方式（恢复原 SQL）。
