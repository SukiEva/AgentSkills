# Intake and decision checklist

只在缺失信息会明显影响判断时继续追问；避免一次性索取过多背景。

## 1. 第一轮必问信息

1. 原始 SQL。
2. 执行计划：优先 `EXPLAIN PERFORMANCE`，其次 `EXPLAIN ANALYZE`。
3. 现象：总耗时、是否偶发、是否最近数据量暴增。

如果以上三项缺失任意一项，优先补齐，再进入深度分析。

## 2. 第二轮按需补问

只有当以下判断会被阻塞时才补问：

### 判断是否适合建索引
- 表是行存还是列存？
- 过滤列、JOIN 键、排序列的基数如何？
- 该 SQL 是否高频执行？

### 判断是否需要重分布/重建表
- 表的当前分布键是什么？
- 高频 JOIN / GROUP BY 键是什么？
- 是否有明显数据倾斜或热点值？

### 判断是否依赖分区裁剪
- 表是否分区？
- SQL 是否按分区键过滤？
- 分区键是否被函数或隐式转换破坏？

## 3. 什么时候停止追问

满足以下任一情况就停止继续追问，先给初步结论：

1. 执行计划已经明确出现 `REDISTRIBUTE` / `BROADCAST` / `GATHER` 大开销。
2. 执行计划已经明确出现大 `Seq Scan` / `CStore Scan` 与低返回行数。
3. 执行计划已经明确出现 DN 严重不均衡或数据倾斜。
4. 用户只想要快速排查方向，而不是最终 DDL 方案。

## 4. 默认分析顺序

1. 先看分布式传输。
2. 再看倾斜。
3. 再看扫描与过滤。
4. 再看 JOIN。
5. 再看聚合 / 排序。
6. 最后看索引。

## 5. 输出建议时的优先级规则

### 高优先级
- 能直接执行且低风险的动作。
- 有明显执行计划证据支撑的动作。
- 能先验证假设的动作，例如 `ANALYZE`、`table_skewness()`、补充 DDL。

### 中优先级
- 需要一定改造但收益较大的动作。
- 例如调整分布键、SQL rewrite、索引方案对比。

### 低优先级
- 缺少关键背景支撑的动作。
- 需要较大表结构改造且当前证据不足的动作。

## 6. 必须避免的输出问题

- 只给“建议加索引”而不解释原因。
- 把分布式瓶颈误判成单机扫描问题。
- 在列存大表场景默认推荐索引。
- 未标注风险就建议重建大表。
- 未标注“需业务确认”就建议可能改变语义的 SQL rewrite。
