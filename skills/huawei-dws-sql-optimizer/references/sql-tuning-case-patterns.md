## 使用规则

1. 先识别主瓶颈：分布式传输 / 倾斜 / 扫描 / 不下推 / 分区裁剪 / JOIN / 聚合排序 / 中间结果。
2. 再匹配最接近的单个案例；必要时允许“一主一辅”两个案例组合。
3. 每次引用案例都要写明：
   - 当前 SQL 与案例的相似证据；
   - 当前 SQL 与案例的关键差异；
   - 为什么优先采用该案例动作，而不是建索引或重写全部 SQL。
4. 默认先给低风险验证，再给高成本改造：
   - 低风险：`ANALYZE`、`EXPLAIN PERFORMANCE`、安全 rewrite、补充 `IS NOT NULL`。
   - 中风险：会话级 GUC、索引试验、partial cluster key。
   - 高成本：改分区、重建大表、重做中间表承载方式；单纯修改分布键时优先先评估直接 `ALTER TABLE ... DISTRIBUTE BY HASH (...)`，不要默认升级为重建表。

### 输出时的案例引用卡片（建议格式）


## 逐案例模式库

### 案例：选择合适的分布列

**优先触发信号**
- `Streaming (type: REDISTRIBUTE)` 是最重节点。
- JOIN / GROUP BY 高频键与现有分布键不一致。
- DN 耗时差异明显，怀疑分布倾斜。

**建议动作**
- 分布列优先满足：高基数、稳定、贴合高频 JOIN / GROUP BY 键。
- 如果当前瓶颈是跨 DN 搬移，优先考虑直接修改分布键，而不是先建索引：

```sql
ALTER TABLE schema.table
DISTRIBUTE BY HASH (new_dist_key);
```

- 只有在直接修改分布键不适用，或还要同时调整分区/存储/其他表结构时，才退回到重建表方案。
- 对中间表或阶段性结果，也要同步检查其分布方式是否破坏后续 colocated join。

**验证方式**
```sql
SELECT table_skewness('schema.table');
EXPLAIN PERFORMANCE <optimized_sql>;
```

**输出提醒**
- 必须说明当前 Streaming 节点预计如何被削弱。
- 必须明确本次建议是“直接 ALTER 修改分布键”还是“重建表改分布”。
- 如果建议直接修改分布键，仍需说明锁、执行窗口、回归验证方式。
- 只有在确有必要时，才说明重建、回灌、窗口期、与其它 SQL 的兼容性。

### 案例：建立合适的索引

**优先触发信号**
- 行存表。
- 高选择性等值过滤后返回很少。
- 最重节点是扫描、回表、排序，而不是广播或重分布。

**建议动作**
- 优先建议复合索引。
- 常见列顺序：等值过滤列 -> 高频 JOIN 列 -> 排序列。
- 如果只是统计信息不准，先 `ANALYZE`，不要把索引当唯一答案。

**验证方式**
```sql
ANALYZE schema.table;
EXPLAIN PERFORMANCE <optimized_sql>;
```

**输出提醒**
- 必须写出建索引原因、字段顺序、写入代价、空间代价、收益预期。
- 对列存大表，默认不把该案例放到高优先级。

### 案例：增加 JOIN 列非空条件

**优先触发信号**
- JOIN 键业务上应为非空，但 SQL 未显式限制。
- 执行计划显示 JOIN 输入包含大量无意义空值探测。
- 过滤和连接无法尽早缩小输入。

**建议动作**
- 在业务语义允许时，为 JOIN 关键列显式补 `IS NOT NULL`。
- 如果表设计允许，也可建议补充约束背景或在 ETL 端消除脏值。

**验证方式**
```sql
EXPLAIN PERFORMANCE
SELECT ...
FROM ...
WHERE join_col IS NOT NULL;
```

**输出提醒**
- 必须标注“需确认业务语义允许忽略空值”。
- 不能把该动作包装成普适优化；它只适用于空值确实无业务意义的场景。

### 案例：典型不下推语句整改案例集

**优先触发信号**
- 谓词、计算、聚合、子查询未在 DN 侧完成。
- CN 或上层节点承担过重过滤/聚合。
- 扫描行数远大于返回行数，且逻辑上本应更早过滤。

**建议动作**
- 优先让过滤、聚合、可等价改写的表达式在 DN 侧完成。
- 检查是否存在函数包裹列、复杂表达式、隐式类型转换、阻断下推的写法。
- 若多个子查询/视图嵌套导致不下推，优先简化层次或前移过滤。

**验证方式**
```sql
EXPLAIN PERFORMANCE <rewritten_sql>;
```

**输出提醒**
- 要明确“当前主要问题是不下推，不是缺索引”。
- 说明优化后预期减少的是扫描量、回传量还是聚合输入量。

### 案例：设置 `cost_param` 对查询性能优化

**优先触发信号**
- 明显怀疑优化器代价模型导致计划选型偏差。
- 同一 SQL 在结构合理时仍稳定选错计划。
- 其它结构性问题已初步排除，但计划仍不理想。

**建议动作**
- 把 `cost_param` 视为验证性调优手段，而不是第一优先级方案。
- 先在会话级验证，再决定是否值得长期固化。
- 必须与执行计划前后对比一起使用。

**验证方式**
```sql
SET cost_param = <value>;
EXPLAIN PERFORMANCE <sql>;
```

**输出提醒**
- 必须标注作用域、回退方法和风险。
- 如果有更直接的结构优化路径，不应先推荐该案例。

### 案例：调整局部聚簇键

**优先触发信号**
- 列存表在特定过滤/排序路径上反复访问。
- 扫描块很多，但热点列具有明显访问局部性。
- 排序、局部有序访问、范围过滤成本高。

**建议动作**
- 优先从高频过滤列、排序列、范围扫描列中选择局部聚簇键候选。
- 结合查询模式判断是否值得为读性能重排局部存储。

**验证方式**
- 对比改造前后的扫描块、排序输入量、总耗时。
- 重新执行 `EXPLAIN PERFORMANCE <sql>;`

**输出提醒**
- 必须说明适用的是列存访问模式，而不是泛化到所有表。
- 如果主要瓶颈仍是重分布或不下推，应先处理那些问题。

### 案例：调整中间表存储方式

**优先触发信号**
- ETL / 报表过程产生大中间结果并被重复扫描。
- 临时表、中间表很宽、很大、落盘重。
- 中间结果承载方式与后续使用模式不匹配。

**建议动作**
- 以批量扫描、聚合为主时，优先评估列存。
- 以小批量点查、频繁更新为主时，再评估行存。
- 同步检查中间表的分布方式，避免只换存储不换分布。

**验证方式**
```sql
EXPLAIN PERFORMANCE <sql_using_intermediate_table>;
```

**输出提醒**
- 要说明中间表是一次性阶段表还是高频复用对象。
- 说明变更后影响的是扫描成本、落盘成本还是后续 JOIN 成本。

### 案例：改建分区表

**优先触发信号**
- 时间或枚举条件很强，但扫描分区仍很多。
- 数据规模已明显超过非分区表适配范围。
- 主要查询模式具有稳定的分区过滤特征。

**建议动作**
- 让分区键与高频过滤模式对齐。
- 在分区设计时考虑数据生命周期、冷热分层和维护成本。
- 如果当前不是“写法破坏裁剪”，而是“设计本身不匹配”，优先考虑该案例。

**验证方式**
```sql
EXPLAIN PERFORMANCE <sql>;
```

**输出提醒**
- 必须区分“分区设计不合理”与“SQL 写法破坏了现有裁剪”。
- 重建分区属于高成本动作，要补充改造窗口和回迁方式。

### 案例：调整 GUC 参数 `best_agg_plan`

**优先触发信号**
- `Aggregate` / `HashAggregate` / 聚合重分布路径明显异常。
- 怀疑局部聚合与全局聚合选择不理想。
- 结构性问题已处理，但聚合计划仍然偏差明显。

**建议动作**
- 把 `best_agg_plan` 当作聚合规划验证手段。
- 会话级试验不同设置，对比执行计划和聚合热点。
- 若聚合前输入量过大，先处理过滤和重分布，再考虑该参数。

**验证方式**
```sql
SET best_agg_plan = on;
EXPLAIN PERFORMANCE <sql>;
```

**输出提醒**
- 必须说明参数切换是试验性动作。
- 必须给出回退方式和适用范围。

### 案例：改写 SQL 排除剪枝干扰

**优先触发信号**
- 分区表理论可裁剪，但计划未裁剪或裁剪很差。
- 分区键上存在函数、表达式、类型转换、复杂条件。
- 逻辑上只查少量分区，但实际扫描范围很大。

**建议动作**
- 把分区键上的函数改写到常量侧。
- 消除破坏裁剪的隐式类型转换和复杂表达式。
- 必要时拆分条件，让分区过滤更直接。

**验证方式**
```sql
EXPLAIN PERFORMANCE <rewritten_sql>;
```

**输出提醒**
- 要明确这是“SQL 写法导致剪枝失效”的案例，不一定需要重建分区表。
- 输出时最好指出哪一段谓词在干扰裁剪。

### 案例：改写 SQL 消除 `in-clause`

**优先触发信号**
- 大 `IN (...)` 列表。
- `IN` 子查询导致计划膨胀、重复半连接或难以下推。
- 查询逻辑可安全改写为 JOIN / EXISTS / 小表驱动。

**建议动作**
- 优先评估改成 `EXISTS`、JOIN 或通过小表承载条件集。
- 如果 `IN` 列表来自应用层拼接，建议转为维表或临时表驱动。

**验证方式**
```sql
EXPLAIN PERFORMANCE <rewritten_sql>;
```

**输出提醒**
- 要说明改写是为降低优化器处理复杂度、减少重复探测或改善下推。
- 必须确认改写不会引入重复行语义变化。

### 案例：使用 `partial cluster key`

**优先触发信号**
- 想利用聚簇收益，但全量 cluster key 成本过高。
- 过滤、排序主要依赖少数关键前缀列。
- 列存扫描与局部排序热点明显。

**建议动作**
- 优先围绕最稳定、收益最高的前缀列评估 partial cluster key。
- 把它视为“局部收益最大化”的折中方案，而不是全面聚簇替代品。

**验证方式**
- 对比关键前缀访问路径的扫描块、排序输入和总耗时。
- 重新查看执行计划中的扫描热点变化。

**输出提醒**
- 需要说明为什么是“partial”而不是完整 cluster key。
- 如果查询模式高度分散，不要强推该案例。

### 案例：`NOT IN` 转 `NOT EXISTS`

**优先触发信号**
- 反连接、排除型子查询很重。
- `NOT IN` 遇到空值语义复杂、去重开销大。
- 计划显示子查询探测成本高或结果不稳定。

**建议动作**
- 优先评估改写为 `NOT EXISTS`。
- 在输出中显式检查子查询列是否可能为 `NULL`。
- 必要时补充去重和空值语义说明。

**验证方式**
```sql
EXPLAIN PERFORMANCE <rewritten_sql>;
```

**输出提醒**
- 必须标注“需确认 NULL 语义”。
- 不能省略语义等价性说明。

## 多案例冲突时的排序规则

如果一个 SQL 同时命中多个案例，按以下顺序确定主案例：

1. **先分布与倾斜**：选择合适的分布列。
2. **再不下推与分区裁剪**：典型不下推语句整改案例集、改写 SQL 排除剪枝干扰、改建分区表。
3. **再 JOIN / 子查询改写**：增加 JOIN 列非空条件、改写 SQL 消除 `in-clause`、`NOT IN` 转 `NOT EXISTS`。
4. **再聚合排序与存储布局**：调整局部聚簇键、使用 `partial cluster key`、调整 GUC 参数 `best_agg_plan`、设置 `cost_param`。
5. **最后才是索引**：建立合适的索引。

原因：DWS 慢 SQL 的大头常先来自分布式传输、扫描量、不下推与裁剪失效；索引通常不是最前面的杠杆。

## 输出约束

1. **案例名称尽量显式写出**  
   例如：`该 SQL 与“案例：改写 SQL 排除剪枝干扰”高度相似。`

2. **每条案例建议都要有证据**
   - 执行计划节点；
   - 行数/耗时偏差；
   - 分区裁剪、Streaming、聚合、扫描等客观信号。

3. **每条建议都要分级**
   - 立即可做；
   - 需验证后执行；
   - 需表设计改造。

4. **凡是可能改语义的动作都要标注**
   - 需业务确认；
   - 需确认 NULL 语义；
   - 需确认重复值/去重语义。

5. **不要把案例当死规则**
   当前 SQL 只要证据不充分，就标注为“高概率推断”。
